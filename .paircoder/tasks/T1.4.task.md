# Objective

Implement account recovery: 16-char alphanumeric code generated at registration (SHA-256+salt stored), recovery endpoint to verify email+code and issue new passkey challenge, old credential invalidated on re-registration.

# Implementation Plan

- `src/lib/recovery.ts`:
  - `generateRecoveryCode()`: `crypto.randomBytes(12)` → base62 → 16 chars (~95 bits entropy)
  - `hashRecoveryCode(code, salt)`: `SHA-256(code + salt)` — NOT bcrypt (spec explicitly calls this out for high-entropy tokens)
  - `verifyRecoveryCode(code, salt, storedHash)`: constant-time compare
- On user registration (T1.3 verify route): generate code, hash, store `recovery_code_hash` + `recovery_code_salt` + `recovery_email`; return code to display once
- `POST /api/auth/recover/verify`: accept `{ email, code }` → match hash → return new WebAuthn registration challenge
- On successful new passkey registration: clear old `passkey_credential_id`, replace with new; regenerate + return new recovery code (single-use replaced)
- `src/app/(auth)/recover/page.tsx`: email + code form → passkey re-enrollment screen

# Acceptance Criteria

- [x] Recovery code: `crypto.randomBytes` → base62 alphabet → 16 chars; stored as `SHA-256(code+salt)`, NOT bcrypt
- [x] `POST /api/auth/recover/verify` matches email + hash; returns new-passkey challenge on success; 401 on mismatch
- [x] On re-registration: old `passkey_credential_id` replaced; new single-use recovery code returned (shown once)
- [x] Recovery code is single-use: hash and salt regenerated after each successful recovery
- [x] "Lost access? Recover account" link visible on `/login` screen

# Verification

- Register user → note recovery code
- Use recovery endpoint with correct code → new passkey challenge returned
- Re-register → new code generated; old code no longer valid
- Use old code → 401
