# Objective

Implement the Teller webhook receiver (HMAC verified), transaction/balance polling cron, Teller Connect enrollment storage, access token encryption, and the full error matrix. Split into 4 files to stay under architecture limits.

# Implementation Plan

- `src/lib/teller/client.ts`: typed HTTP client for Teller REST API; handles `Authorization: Basic <token>` header; `getTransactions(accountId)`, `getBalances(accountId)`, `getAccounts()`
- `src/lib/teller/errors.ts`: error matrix — 401 → set `teller_sync_failed` flag + reconnect; 429 → exponential backoff with jitter (max 1hr before notifying); 500 → 3× retry then flag; disconnect → preserve data + flag
- `src/lib/teller/polling.ts`: polling logic called by cron route; deduplicates transactions by `teller_transaction_id`
- `src/app/api/webhooks/teller/route.ts`: HMAC signature verification with `TELLER_SIGNING_SECRET`; dispatch `transaction.created` to classification pipeline
- `src/app/api/cron/teller-poll/route.ts`: Vercel cron handler; calls polling.ts; protected by `CRON_SECRET` header
- Access token: encrypt with AES-256-GCM before storing; decrypt in client.ts for each request
- `vercel.json`: add cron config for `teller-poll` (every 30 min) and balance poll (every 4 hours)

# Acceptance Criteria

- [x] `POST /api/webhooks/teller` verifies HMAC; rejects unsigned requests with 401
- [x] Webhook stores raw transaction to DB then enqueues classification (calls T2.3 pipeline)
